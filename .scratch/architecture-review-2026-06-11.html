Problem: Every bug fix or feature requires editing two copies in different idioms — TypeScript with imports vs JS string template without types. Already drifted: 5+ known behavioural differences.

Solution: Write the browser client once as TypeScript. Build it to JS with tsup/esbuild. Vite plugin reads the built file as a string for its virtual module. Next.js imports it as a normal module.

Wins

  • Locality: one place to fix bugs (currently 5+ drift bugs)
  • Leverage: one source, two adapters (Vite + Next.js)
  • Interface shrinks: vite-plugin.ts drops from 770 → ~80 lines
  • Type safety: the built output was TypeScript-checked before stringification
  • Delete ~720 lines of untyped, unlintable template string

Unify parseDataOid into one canonical return type

Strong in-process
packages/react-plugin/src/client.ts parseDataOid() → { filePath, line, column }
packages/react-plugin/src/vite-plugin.ts parseDataOid() → { marker, projectHash, relativeMarker, filePath, line, column }
packages/react-plugin/src/data-oid.ts parseDataOid() → { type, projectHash, filePath, line, column }
packages/extension/src/data-oid.ts (same as above)

Before — three return shapes

client.ts
{ filePath, line, column }
// drops type + projectHash
vite-plugin.ts
{ marker, projectHash, relativeMarker, filePath, line, column }
// different keys, extra field
data-oid.ts (canonical)

// full shape, but browser copies don't use it

After — one canonical type

data-oid.ts (single source)
DataOidComponents { type, projectHash, filePath, line, column }
Browser consumers
const { filePath, line } = parseDataOid(oid)!
// destructure what you need
Server consumers
const { filePath, line, column } = parseDataOid(oid)!
// same type, different projection

Problem: Same function, three different return types. Renaming a field requires editing N copies and hoping consumers don't break.

Solution: One parseDataOid in data-oid.ts returning DataOidComponents. Browser code destructure-projects what it needs. If Candidate 1 is adopted, this unification is automatic — there's only one browser copy.

Wins

  • Locality: one parse function, one return type
  • Delete 2 copies of parseDataOid with wrong/different shapes
  • Eliminates key-name confusion: marker vs type vs nothing

Note: If Candidate 1 is adopted, the Vite plugin's inlined parseDataOid disappears (replaced by the built output from the shared source). Candidate 3 becomes a smaller step: just move data-oid.ts to a shared location and re-export. Still worth doing, but lower priority.

Top recommendation

Start with Candidate 1: Collapse the browser client

It subsumes Candidates 2 and 3. When the browser client is one TypeScript source, there's one parseDataOid (fixing C2) and the vite-plugin.ts template string disappears (fixing 5 drift bugs at once). C3 (data-oid.ts across packages) is then a small follow-up — move the file to a shared location.

The mechanism: write client.ts once. Build with tsup to dist/browser-client.js. Vite plugin reads the built file with readFileSync() and serves it as a virtual module. Next.js imports it as a normal ESM module via next.tsx. Both paths get the same TypeScript-checked, linted, IDE-navigable code.
