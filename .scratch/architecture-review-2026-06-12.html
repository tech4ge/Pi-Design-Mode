Problem: The previous review collapsed two copies into one source of truth — good. But the resulting browser-client.ts is a 697-line god file. All 8 mutable state variables (selections, ws, isAltDown, isProcessing, isConnected, lastSelections, submittedOids, widgetHost) are in a single closure scope. Every function sees every variable. The createWidget() function alone is 200+ lines of CSS + HTML + event wiring inside a single function body. Applying the deletion test: if you delete this file, all the complexity reappears elsewhere because it cannot be split — there are no seams inside.

Solution: Extract logical groups into modules that export factory functions. The top-level IIFE becomes a thin orchestrator: create the connection, create the selection manager, create the widget, wire them together. Each sub-module closes over only the state it needs (locality). The orchestrator defines the seam: what flows between sub-modules via explicit parameters, not shared closure variables.

Wins

  • Locality: each module owns its state — e.g. selection.ts owns the selections array, not a shared closure
  • Testability: extract selection, history, hover-tooltip as pure-ish functions testable with a DOM fixture. Currently zero behavioral tests for 697 lines
  • Leverage: history.ts and hover-tooltip.ts have tiny interfaces — high depth. Widget render logic is isolated, not interleaved with WS reconnection
  • Refactor safety: change CSS layout without touching selection persistence. Change reconnection strategy without touching widget rendering

Constraint: All sub-modules are built by tsup into a single IIFE dist/browser-client.js. They cannot be separate files at runtime — they must be inlinable. The factory-function pattern (each module exports a function, tsup bundles the calls) works here. This is not about npm packages — it's about internal seams inside one build artifact.

Problem: data-oid.ts mixes browser-safe functions (parseDataOid, formatDataOid) with a Node-only function (hashProjectRoot → node:crypto). The browser client can't import from data-oid.ts because tsup would inline the crypto dependency into the IIFE bundle (which runs in the browser). So it re-implements parseDataOid inline — the same 15 lines, same logic, now maintained twice.

Solution: Split data-oid.ts into data-oid/shared.ts (browser-safe: parseDataOid, formatDataOid, DataOidComponents) and data-oid/index.ts (re-exports + hashProjectRoot with node:crypto). The browser client imports from data-oid/shared — tsup inlines only the browser-safe code. All other consumers import from data-oid unchanged.

Wins

  • Locality: one parseDataOid, one DataOidComponents type
  • Delete ~15 lines of redundant inline code in browser-client.ts
  • Future-proof: adding a new OID format requires editing one file, not two
  • No package boundary change: existing @pi-design/react-plugin/data-oid export continues to work

Fix PiDesignClient — dynamic import without effect guard

Strong bug
packages/react-plugin/src/next.tsx import() inside render body — no useEffect

Before

"use client";

export function PiDesignClient() {
  if (process.env.NODE_ENV !== "production") {
    import("@pi-design/react-plugin/browser-client");
  }
  return null;
}

After

"use client";
import { useEffect } from "react";

export function PiDesignClient() {
  useEffect(() => {
    if (process.env.NODE_ENV !== "production") {
      import("@pi-design/react-plugin/browser-client");
    }
  }, []);
  return null;
}

Problem: The dynamic import() runs on every render of PiDesignClient, not just on mount. While the module is cached and won't re-fetch, a new Promise is created each time the parent re-renders. Worse, the if (NODE_ENV) guard evaluates on every render, creating a subtle timing difference between development and production. The component returns null and has no hook lifecycle — it's a pure side-effect component with no effect boundary.

Solution: Wrap the import in useEffect(() => { ... }, []). Runs once on mount. Idiomatic React. No behavioral change — the IIFE guards against double-init with __piDesignInit — but eliminates the per-render import call and makes the intent clear.

Wins

  • Correctness: side effect runs once on mount, not per-render
  • Clarity: useEffect signals "this component exists for its side effect" — React convention
  • 1 line changed — tiny fix, high clarity

Problem: The WS protocol is the primary contract between extension and browser client. Currently, server.ts defines typed discriminated unions (ClientMessage, ServerMessage), but the browser client constructs and sends messages as plain objects with no type checking. If someone adds a new message type or changes a field name in the server, the client won't break at build time — only at runtime when messages are silently catch'd away.

Solution: Extract protocol types into a browser-safe protocol.ts (pure TypeScript interfaces, no Node imports). Both server and browser-client import from it. The browser client's sendMessage function gets typed. When the protocol changes, both sides fail to compile if they drift.

Wins

  • Leverage: protocol types are the deep seam — both sides depend on them
  • Compile-time drift detection: adding/changing a message type breaks both builds, not just one
  • AI-navigability: "what messages does the client send?" → look at the protocol, not grep through untyped send calls

Cost: extra file + export path. The types add zero runtime code — they're erased by the IIFE build. But they do create a new import the browser-client build must resolve. If Candidate 2 is adopted (splitting data-oid.ts), this follows the same pattern: a browser-safe sub-module that tsup inlines.

Add WS reconnection backoff + max attempts

Worth exploring robustness
packages/react-plugin/src/browser-client.ts ws.onclose → setTimeout(connectWS, 2000) — fixed 2s retry, infinite, no backoff

Problem: On WS close, the browser client retries every 2 seconds forever with no backoff. If the design mode server is stopped, this creates an infinite stream of failed connection attempts — logged to the console, consuming resources, and never resolving. The user sees a "Connection lost" error that never clears.

Solution: Exponential backoff (2s → 4s → 8s → 16s, cap at 30s) with a max attempt count (e.g. 10). After exhausting retries, show a permanent "Disconnected — run /design to restart" message and stop retrying. The user can still manually reload the page.

Wins

  • Locality: reconnection strategy is in one place (the connectWS function) instead of being implicit in the retry loop
  • Resource hygiene: no infinite retry storm when design mode is off
  • User experience: clear terminal state instead of perpetual "will retry"
